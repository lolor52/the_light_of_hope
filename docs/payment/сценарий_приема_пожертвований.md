# Сценарий приема пожертвований

## Зафиксированное ограничение

Проект использует только чистый frontend.

В рамках сайта запрещены:

- backend;
- serverless-функции;
- база данных;
- хранение секретов ЮKassa в коде сайта.

## Допустимые сценарии оплаты

### Вариант 1. Только СБП

Сайт может использовать только СБП.

Корректный сценарий для этого ограничения — размещение на сайте статического QR-кода СБП, который пользователь открывает в банковском приложении.

Что это дает:

- полностью совместимо с чистым frontend;
- не требует backend;
- не требует базы данных;
- подходит для быстрого запуска приема пожертвований.

Что это не дает:

- сайт не может достоверно определить, что платеж успешно завершен;
- сайт не может автоматически создавать уникальные платежи под конкретного жертвователя;
- сайт не может надежно хранить статус оплаты как юридически значимый факт.

### Вариант 2. Внешняя страница ЮKassa

Сайт может вести пользователя на заранее созданную ссылку оплаты или на счет, созданный вне сайта.

Что важно:

- сайт только перенаправляет пользователя;
- ссылка должна быть создана заранее в ЮKassa вне frontend-кода сайта;
- сам сайт не создает платеж через API.

Ограничение:

- если нужна динамическая сумма, описание платежа или отдельный платеж под каждого пользователя, этот вариант уже выходит за рамки чистого frontend.

## Рекомендуемый вариант для текущего проекта

Если придерживаться требования «только чистый frontend», базовый и устойчивый вариант — использовать только СБП через готовый QR-код.

Этот вариант соответствует ограничениям проекта и не требует внедрения backend.

## Практическое правило для дальнейших изменений сайта

Любые изменения блока пожертвований должны соответствовать одному из двух сценариев:

- статический QR-код СБП;
- внешняя заранее созданная ссылка на оплату ЮKassa.

Если задача требует хотя бы одного из пунктов ниже, она уже не относится к чистому frontend и должна быть отклонена в рамках этого проекта:

- создание платежа по API;
- получение или проверка статуса платежа в реальном времени;
- подтверждение успешной оплаты на стороне сайта;
- безопасная авторизация плательщика по данным оплаты.
