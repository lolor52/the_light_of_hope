# Ликвидность — расчёт за Основную сессию MOEX

> Все расчёты выполняются **только по Основной торговой сессии MOEX**. Вечерняя сессия и все аукционы исключены.

## Входные параметры
```
Ticker = <код тикера>
Date   = <YYYY-MM-DD> (MSK)
S      = множество тикеров для кросс-нормировки на Date (включая Ticker)
```
> Если метрика считается для одного тикера, множество `S` всё равно задаётся для нормировки; при |S|=1 используйте защиту `ε` в формулах нормировки.

## Источник данных
Только **MOEX ISS HTTP API** (`iss.moex.com`). Сторонние источники и стримы не используются.
Используемые ресурсы:
- `securities` — справочник по бумагам (в т.ч. `LOTSIZE`, `MINSTEP`).
- `candles` с `interval=1` — минутные свечи за дату.
- `boards/schedule` — расписание торгов для определения границ Основной сессии и аукционов.

## Подготовка данных
1) Определите границы Основной сессии по `boards/schedule` для выбранного инструмента и даты `Date`. Исключите интервалы аукционов открытия/закрытия.
2) Запросите `candles` (`interval=1`) целиком на `Date`. Оставьте только минуты, попадающие в Основную сессию. Минуты с нулевым оборотом допустимы.
3) Из `securities` возьмите `LOTSIZE` и `MINSTEP` для `Ticker`.
4) Для каждой минуты *m* сохраните поля `open_m, high_m, low_m, close_m, volume_m, value_m`.
   - Если `value_m` недоступно, используйте прокси:  
     `value_m := volume_m * LOTSIZE * price_proxy_m`, где `price_proxy_m := (high_m + low_m + 2*close_m)/4`.
5) Обозначьте множество минут Основной сессии как `M`. Пусть `M_act = { m ∈ M | volume_m > 0 }`.

Константы: `ε = 1e-12` (числовая защита). Время — MSK (UTC+3).

## Минутные величины
- Ретёрн по минуте: `r_m = | ln( close_m / close_{m-1} ) |`, если предыдущая минута отсутствует или `close_{m-1} ≤ 0`, то пропустите такую пару.
- Изменение цены: `Δp_m = close_m - close_{m-1}` (аналогично пропускать при отсутствии пары).
- Базовая цена для тик-процента: `price_ref = median( { close_m | m ∈ M_act } )`.

## Сессионные метрики (только из `candles` + `securities`)
1) **Суммарный денежный оборот**:  
   `V_total = Σ_{m∈M} value_m`.
2) **Медианный минутный оборот по активным минутам**:  
   `V_med = median( { value_m | m ∈ M_act } )` (если `M_act` пусто, метрика отсутствует).
3) **Доля активных минут**:  
   `A = |M_act| / |M|`.
4) **Amihud Illiquidity (1m)**:  
   `ILLIQ = mean( { r_m / (value_m + ε) | m ∈ M_act и r_m определён } )`.
5) **Roll spread proxy (1m)**:  
   `R = 2 * sqrt( max( 0, - cov(Δp_m, Δp_{m-1}) ) )`, ковариация по множеству минут, где определены обе разности.
6) **Tick% (крупность тика)**:  
   `TickPct = 10000 * MINSTEP / price_ref`.
7) **Прокси глубины**:  
   `DepthProxy = mean( { value_m / (r_m + ε) | m ∈ M_act и r_m определён } )`.

> Если размер выборки для любой метрики < 30 минут, пометьте её как «низкая надёжность», но расчёт допускается.

## Нормировка метрик по срезу S в дату Date
Для каждой метрики `X ∈ { V_total, V_med, A, ILLIQ, R, TickPct, DepthProxy }` рассчитайте по всем инструментам из `S` в ту же дату следующие квантили:
`P5_X = 5-й перцентиль`, `P95_X = 95-й перцентиль`.
Определите функции нормировки в [0,1]:
```
norm_up_X(x)   = clip( (x - P5_X) / (P95_X - P5_X + ε), 0, 1 )
norm_down_X(x) = 1 - norm_up_X(x)
```
При |S|=1 используйте `P5_X = x`, `P95_X = x`, тогда `norm_*` дадут 0.5 по определению.

## Итоговый скор ликвидности
Весовая схема (сумма весов = 1):
```
w_Vtotal = 0.30
w_Vmed   = 0.20
w_Depth  = 0.20
w_A      = 0.10
w_ILLIQ  = 0.10   (через norm_down)
w_Roll   = 0.07   (через norm_down)
w_Tick   = 0.03   (через norm_down)
```
Расчёт:
```
L_raw = w_Vtotal * norm_up_V_total(V_total)
      + w_Vmed   * norm_up_V_med(V_med)
      + w_Depth  * norm_up_DepthProxy(DepthProxy)
      + w_A      * norm_up_A(A)
      + w_ILLIQ  * norm_down_ILLIQ(ILLIQ)
      + w_Roll   * norm_down_R(R)
      + w_Tick   * norm_down_TickPct(TickPct)

L = round( 100 * clip(L_raw, 0, 1), 2 )
```
`L ∈ [0, 100]`.

## Правила исключений и обработка отсутствующих данных
- Если `|M| < 60` минут после фильтрации по сессии, верните `L = NaN` и статус `insufficient_session`.
- Если для метрики нет значений, используйте среднее по S после нормировки: `0.5` и отметку `imputed`. Импутации фиксируются в отчёте.
- Свечи, попадающие в интервалы аукционов по `boards/schedule`, полностью исключаются из `M`.
- Никаких данных вне MOEX ISS не используется. Тик-данные, стакан, сделки со стороны агрессора не требуются.

## Псевдозапросы ISS (пример)
- `.../iss/engines/<engine>/markets/<market>/boards/<board>/securities/<Ticker>.json?iss.only=securities`
- `.../iss/engines/<engine>/markets/<market>/boards/<board>/securities/<Ticker>/candles.json?from=Date&to=Date&interval=1`
- `.../iss/engines/<engine>/markets/<market>/boards/<board>/sessions.json` или `.../boards/<board>/schedule` для календаря основной сессии.
Точные пути зависят от рынка/борда; используйте те, что документированы в ISS для бумаги.

## Параметры
```
ε = 1e-12
MainSessionOnly = true
ExcludeAuctions = true
TZ = MSK (UTC+3)
Quantiles = (5%, 95%)
```
## Выход
Одно число `L(Ticker, Date) ∈ [0,100]` и при необходимости диагностический отчёт: размеры выборок, доля импутаций, флаги надёжности.
